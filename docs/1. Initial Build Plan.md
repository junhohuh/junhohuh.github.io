# Personal website: build plan and shared AI context

Source of truth for original requirements: [docs/PROJECT_INTAKE.md](PROJECT_INTAKE.md). In Cursor, reference it as `@docs/PROJECT_INTAKE.md` (or `@PROJECT_INTAKE.md` if you open that file from the repo).

**Chosen approach (MVP):** **Astro** + TypeScript + **Tailwind CSS**, **static output** only (pre-built HTML/CSS/JS in `dist/`; no server, database, or auth). Record the exact deploy target in [docs/ARCHITECTURE.md](ARCHITECTURE.md) and log the framework choice in [docs/DECISIONS.md](DECISIONS.md).

---

# 1. Project Summary

- **What the project is:** A public personal website with a landing page, short bio and contact info, a projects showcase, and a clear path to your resume—optimized for web and mobile browsers.
- **Who it is for:** You as the owner/editor; **readers** are recruiters and hiring managers who open your link from applications.
- **Why it matters:** It is a single, professional URL that presents who you are, what you have built, and how to reach you—structured enough to scan quickly, polished enough to trust.

---

# 2. MVP Scope

**Core features (from intake):**

1. Landing page that orients visitors and introduces you.
2. About/bio plus contact information (email, LinkedIn, GitHub—or whatever you choose to publish).
3. Resume access (Highlight key sections of my resume on the website, but also include full resume link)
4. Projects section (each project: name, one-line summary, tech, link to repo/demo if any).
5. Clean visual polish: readable typography, consistent spacing, responsive layout, professional tone with a bit of personality.

**Non-goals (explicit):**

- Native mobile apps (per intake).
- For MVP: no blog/CMS, no comments, no login, no admin dashboard, no database—unless you later choose a static CMS (see Future).

**Success criteria:**

- All five MVP items work on **desktop and mobile browsers**.
- A recruiter can understand **who you are**, **what you have built**, and **how to contact you** in under ~2 minutes.
- Lighthouse/accessibility is “good enough” for a student portfolio (no broken links, readable contrast, keyboard-navigable).

**Future ideas (post-MVP):**

- Hobbies Section
- Blog or “writing” section; MDX/Markdown-driven project pages; dark mode; analytics (privacy-respecting); contact form (needs backend or third-party form service); CMS (Contentlayer, Sanity, etc.); custom domain + email.

# 3. Recommended Stack


| Layer                  | Recommendation                                                                 | Why                                                                                                                                 |
| ---------------------- | -------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **Frontend**           | **Astro + TypeScript + Tailwind CSS**                                            | Built for mostly-static sites; ships little JS by default; great for portfolio pages and optional MD/JSON-driven content.             |
| **Build / output**     | **Static (`output: 'static'`)** — Astro default                                  | Produces a folder of files (`dist/`) any static host can serve; no Node server required in production for MVP.                      |
| **Backend**            | **None for MVP**                                                                 | Keeps deploy and security simple.                                                                                                  |
| **Database**           | **None for MVP**                                                                 | Projects and copy live in-repo (e.g. `.astro`, `.md`, or `src/content/`).                                                           |
| **Auth**               | **None**                                                                         | Public site only.                                                                                                                   |
| **Hosting**            | **GitHub Pages** and/or **Netlify** / **Vercel** (all work with static `dist/`) | *Assumption:* you use GitHub for the repo. GitHub Pages fits a pure static site; Netlify/Vercel are equally fine with Astro builds. |
| **AI / external APIs** | **None required**                                                                | Add later only if you add a contact form, analytics, or CMS.                                                                        |


**Alternative (if you later need heavy React + server features):** **Next.js on Vercel**—not needed for this MVP.

# 5. Shared Context / Repo Memory Setup

**Goal:** One canonical story of the project; tool-specific files only **point** and **constrain**, they do not fork the truth.

**Markdown files to add**


| File                                               | Responsibility                                                                                                                                                      | Long-term memory?                                                  |
| -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| **[README.md](../README.md)**                         | Human-first: what the repo is, how to run/build/deploy, link to `docs/`.                                                                                            | Partially—keep it short; deep detail lives in `docs/`.             |
| **[docs/PROJECT_BRIEF.md](PROJECT_BRIEF.md)** | Product intent: audience (recruiters), MVP scope, non-goals, tone (professional + a little fun), success criteria. **Promote content from [PROJECT_INTAKE.md](PROJECT_INTAKE.md) here.** | **Yes—primary canonical “why.”**                                   |
| **[docs/ARCHITECTURE.md](ARCHITECTURE.md)**   | Stack, folder map, routing, where content lives (e.g. `src/data/projects.ts` or `src/content/`), deploy target, env vars (if any).                                     | **Yes—canonical “how it’s built.”**                                |
| **[docs/TASKS.md](TASKS.md)**                 | Checklist: now / next / done; single place for “current state.” Update when you finish a session.                                                                   | **Yes for “now”;** archive done items periodically to avoid noise. |
| **[docs/DECISIONS.md](DECISIONS.md)**         | Short dated entries: “Chose X over Y because …” (1–5 lines each).                                                                                                   | **Yes—prevents re-litigating stack.**                              |


**Reduce drift between tools**

1. **Single source of truth:** `PROJECT_BRIEF.md` + `ARCHITECTURE.md`. Do not paste full specs into three places.
2. **Thin tool entrypoints:** `CLAUDE.md`, `GEMINI.md`, and `.cursor/rules` should say: read `docs/PROJECT_BRIEF.md` and `docs/ARCHITECTURE.md` first; follow `docs/TASKS.md` for current work.
3. **When you change stack or structure:** update `ARCHITECTURE.md` + one line in `DECISIONS.md` the same day.
4. **Weekly (or per milestone):** trim `TASKS.md` done section or move to a “CHANGELOG” snippet at bottom.

**What goes in CLAUDE.md / GEMINI.md / Cursor Rules**

- **CLAUDE.md** (repo root, Claude Code): 15–30 lines max: project one-liner; `npm`/`pnpm` dev and build commands (`astro dev`, `astro build`, `astro preview`); “read `docs/` first”; coding conventions (TypeScript strict, Tailwind + `.astro`, no new deps without DECISIONS entry); “do not expand MVP scope without updating PROJECT_BRIEF.”
- **GEMINI.md:** Same content as `CLAUDE.md` (or literally: “Same as CLAUDE.md; read docs/PROJECT_BRIEF.md and docs/ARCHITECTURE.md.”) so two tools never diverge on instructions.
- **Cursor Rules** (e.g. `.cursor/rules/personal-website.mdc`): Scoped rules—match existing code style, prefer small PR-sized changes, list forbidden patterns (e.g. no `any`, no inline secrets). Optional second rule file for `src/**/*.astro` and `src/components/**` if UI patterns grow.

**PROJECT_INTAKE.md**

- Raw intake lives in [docs/PROJECT_INTAKE.md](PROJECT_INTAKE.md). After you add `docs/PROJECT_BRIEF.md`, either keep the intake as historical reference only or add a one-line note at the top of the brief: “Derived from docs/PROJECT_INTAKE.md” so tools do not treat competing narratives as equal without context.

# 6. Key Technical Decisions (before coding)

1. **Framework:** **Astro (static)** — decided for this repo; log date + rationale in `DECISIONS.md` (e.g. static portfolio, minimal JS, simple deploy).
2. **Hosting:** **GitHub Pages** vs **Netlify/Vercel** — any works with `dist/`; if using a **project** site (`username.github.io/repo-name/`), set Astro `base` correctly so assets resolve. Record choice in `ARCHITECTURE.md`.
3. **Resume delivery:** PDF in `public/` vs external link (Drive/Dropbox)—PDF-in-repo is simplest and always works with static hosting.
4. **Projects data shape:** JSON/TS module vs Astro **content collections** (MD/MDX)—JSON or a single `projects` module is fastest for MVP; collections help when you add many write-ups later.
5. **Typography / UI baseline:** One font pairing + a simple color system (Tailwind theme or CSS variables)—decide early for “clean polish.”
6. **Package manager:** npm vs pnpm—pick one, document in README and ARCHITECTURE.
7. **Domain:** `github.io` / host default subdomain for MVP vs custom domain—custom can wait; note in BRIEF if you plan it.

# 7. Risks / Unknowns

- **Scope creep:** AI assistants often suggest blogs, CMS, auth—guard with `PROJECT_BRIEF.md` non-goals and Cursor rules.
- **Duplicate “intake” files:** Keeping both `docs/PROJECT_INTAKE.md` and `PROJECT_BRIEF.md` without a clear relationship (derived vs canonical) causes conflicting instructions.
- **Stale TASKS.md:** If not updated, tools will implement the wrong “current” task—make “update TASKS” part of your shutdown ritual.
- **Resume privacy:** Public repo + PDF in `public/` is world-readable; if the resume must stay semi-private, use a password-protected or unlisted host and link only in the site (still a tradeoff).
- **Over-polishing:** Two-to-three weeks can disappear into animation libraries—cap MVP polish to layout, type, and responsiveness first.
- **GitHub Pages + subpaths:** If the site is not at the root of a `username.github.io` repo, misconfigured `base` in `astro.config` breaks CSS and links—document the final URL shape in `ARCHITECTURE.md`.

# 7½. Scaffold Review (2026-03-22)

Post-scaffold audit comparing the initial code against this build plan. **Build passes cleanly.** Structure, docs, and stack all match the plan. The items below are the only gaps found.

### Inconsistencies

1. **Footer external links missing `target="_blank"` / `rel="noopener noreferrer"`** — GitHub and LinkedIn links in `Footer.astro` open in the same tab, unlike every other external link in the scaffold (Header Résumé link, links in `about.astro`, `index.astro`). Should be consistent.

2. **Inconsistent inline-link styling within `about.astro`** — The email link (line 26) uses underline decoration classes (`underline decoration-teal-800/25 underline-offset-2 hover:decoration-teal-800`), but the résumé PDF link (line 40) in the same page does not. Both are inline text links and should share the same visual treatment.

3. **No explicit `output: 'static'` in `astro.config.mjs`** — §3 and §6.1 of this plan reference `output: 'static'`. It *is* the Astro default, so omitting it works, but adding it explicitly documents intent and keeps the config in sync with what `ARCHITECTURE.md` describes.

### Improvements (not blocking, but worth addressing)

4. **No custom font pairing chosen** — §6.5 recommends deciding on "one font pairing + a simple color system" early. The color system (slate + teal accent) is solid, but fonts fall back to browser defaults. Log the decision (system stack vs. a chosen web font) in `DECISIONS.md` so it's intentional rather than forgotten.

5. **`{' '}` whitespace syntax in `about.astro`** — Lines 23 and 39 use JSX-style `{' '}` for spacing. This works in Astro templates but is unnecessary—regular whitespace is preserved in `.astro` files. Minor idiomatic cleanup.

6. **`.vscode/` directory tracked in git but absent from planned structure (§4)** — Contains `extensions.json` (Astro extension recommendation) and `launch.json` (dev server launch config) from `create astro`. Harmless, but either acknowledge it in `ARCHITECTURE.md` or add it to `.gitignore` to keep the repo structure aligned with the plan.

7. **New-tab links lack assistive-technology indication** — Links with `target="_blank"` (résumé PDF) don't tell screen readers they open a new window. Step 8 calls for "basic a11y." Adding "(opens in new tab)" via `aria-label` or visible text is a small win.

### Confirmed OK

- All five MVP pages/features from §2 are implemented (landing, about + contact, résumé highlights + PDF link, projects with data-driven cards, responsive polish).
- Docs ecosystem (§5): `PROJECT_BRIEF.md`, `ARCHITECTURE.md`, `TASKS.md`, `DECISIONS.md`, `CLAUDE.md`, `GEMINI.md`, `.cursor/rules/personal-website.mdc` all exist and are internally consistent.
- Stack matches §3 exactly: Astro 6, TypeScript strict, Tailwind 4 via `@tailwindcss/vite`, static output, no backend/DB/auth.
- Projects data uses a TS module (`src/data/projects.ts`) per §6.4 "fastest for MVP" path.
- Skip-to-content link in `Layout.astro` and ARIA landmarks are in place.
- `.gitignore` covers `dist/`, `.astro/`, `node_modules/`, `.env`, `.DS_Store`, `.venv/`.

# 8. First Build Plan (5–10 steps)

1. Initialize git repo; scaffold **Astro** (`npm create astro@latest`) with **TypeScript**; add **Tailwind** via `astro add tailwind`; confirm `astro build` produces **`dist/`** and `astro preview` looks correct.
2. Ensure `docs/PROJECT_BRIEF.md`, `docs/ARCHITECTURE.md`, `docs/TASKS.md`, `docs/DECISIONS.md` exist; add thin `CLAUDE.md`, `GEMINI.md`, `.cursor/rules`; add minimal `README.md` with dev/build/preview commands. Log **Astro + static** in `DECISIONS.md`.
3. Implement shared **Layout** (`src/layouts/Layout.astro`): header/nav (Home, About, Projects, Resume), footer with contact links.
4. Build **Landing** (`src/pages/index.astro`): headline, subline, primary CTAs to About / Projects / Resume.
5. Build **About** (`src/pages/about.astro`): short bio + contact block (consistent with footer); include resume highlights if desired (per MVP).
6. Add **projects** data (e.g. `src/data/projects.ts` or `src/content/`) and **Projects** (`src/pages/projects.astro`) with cards (title, summary, tech tags, links).
7. Add **resume**: place `resume.pdf` in `public/` and wire a prominent Resume nav link + download/open link.
8. Responsive pass + basic a11y (landmarks, focus states, alt text if images).
9. Deploy **static `dist/`** to your chosen host (GitHub Actions → Pages, or Netlify/Vercel); fix any **base URL** or asset path issues; smoke-test on mobile.
10. Final pass: proofread copy, check all links, update `TASKS.md` and `DECISIONS.md` with “MVP shipped” and deploy notes.
